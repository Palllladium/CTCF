# Phase 12-A: TTO — инструкция по запуску

## Что это и зачем

Каскад CTCF выдаёт поле деформации за один прямой проход. **TTO (test-time optimization)** —
это дооптимизация **самого поля** под конкретную пару изображений, уже после того как сеть отработала.
**Веса модели заморожены, градиент к ним не идёт вообще.** Это не дообучение, а уточнение на инференсе.

Зачем: ось «архитектурных улучшений» упёрлась в потолок 0.832–0.833 (Phase 11). TTO — **другая ось**:
она даёт неограниченную ёмкость на каждую пару отдельно. Ожидаемый прирост +0.008…+0.025 Dice.

**Всё ниже — только инференс. Никакого обучения. ~20 часов суммарно** против 69 часов за один
500-эпошный прогон.

---

## Запуск

```bash
cd <repo>
git pull                      # нужна ветка ctcf-p3: utils/tto.py, models/KAN_IDIR/

# всё сразу (~20 ч, блоки идут по убыванию важности)
bash tools/runners/tto_ladder.sh

# или по одному блоку
BLOCK=A bash tools/runners/tto_ladder.sh
BLOCK=C bash tools/runners/tto_ladder.sh
```

Скрипт **резюмируемый**: прогон, у которого уже есть `summary.csv`, пропускается. Можно спокойно
прерывать и перезапускать. Если чекпоинт не найден — печатает `[MISS]` и идёт дальше, не падает.

Чекпоинты берутся оттуда, куда их пишет наш собственный обучающий код:
`results/<EXP_NAME>/ckpt/best.pth`. Ничего перекладывать не надо.

## Блоки

| блок | что отвечает | прогонов | время* |
|---|---|---|---|
| **A** | Работает ли TTO и какая параметризация лучше (`svf`/`disp`/`inr`/`kan`/`randkan`) | 6 | ~4 ч |
| **B** | Не является ли результат просто удачно/неудачно подобранным learning rate | 3 | ~2 ч |
| **C** | **Главный.** Схлопывает ли TTO «лестницу» из 4 чекпоинтов Phase 11 | 4 | ~3 ч |
| **D** | **Доменная адаптация:** заменяет ли «один чекпоинт + TTO» чекпоинт под датасет | 4 | ~6 ч |
| **E** | Центральный тезис статьи: держит ли SVF топологию **без** якобианова штрафа | 2 | ~1.5 ч |
| **F** | Окупаются ли два приёма из KAN-IDIR (ε-запас, маска мозга) | 2 | ~1.5 ч |
| **G** | Нативный IXI (**самый дорогой**, 115 пар — последним) | 2 | ~5 ч |

\* оценка по замеру на RTX 5070 (160×192×224): `svf` 0.73 с/шаг, `disp` 0.57, `inr` 1.10.
На RTX PRO 6000 должно быть примерно вдвое быстрее. Пик памяти самой TTO — **3.2 GB**.

### Блок A — первый и обязательный

Первый прогон `A_HEAD__none` — это **контроль**: TTO выключен, модель должна выдать
**ровно 0.8314** (наше опубликованное число для Mamba SVF 500ep).
**Если контроль не сошёлся — остановитесь и напишите нам, дальше бессмысленно.**

Режимы `kan` и `randkan` — это **сети из KAN-IDIR** (лицензия MIT), вставленные как наш остаток.
Это даёт прямое сравнение «их сеть, стартующая с нашего каскада» против «их сеть, стартующая
с тождественного отображения» — то есть ровно тот вклад, который мы заявляем.

### Блок C — ради чего всё затевалось

Четыре чекпоинта Phase 11 стоят на четырёх уровнях Dice: 0.8257 / 0.8288 / 0.8296 / 0.8305.
- Если TTO поднимает их **все в одну точку** → архитектурные улучшения покупали ровно то, что
  TTO и так даёт → **L3CH64 не нужен, и 69-часовой прогон STACK@500ep можно не делать вообще.**
- Если «лестница» сохраняется → ёмкость комплементарна TTO, за неё надо платить.

Базовые числа (без TTO) для этих четырёх уже есть в `results/SEDM/inference/` — перезапускать не нужно.

### Блок D — доменная адаптация

Чекпоинт, обученный на OASIS, гоняется на IXI, и наоборот. **Веса новый датасет не видят вообще** —
переоптимизируется только поле, по каждой паре, без меток. Вопрос: закрывает ли TTO просадку
от смены домена. Если да — «один чекпоинт + TTO ≈ чекпоинт под конкретный датасет».

## Что прислать обратно

Всю папку **`results/tto/`** целиком. Там только CSV/JSON, несколько мегабайт:

```
results/tto/<run>/per_case.csv     # по паре: Dice, sdlogj, время (та же схема, что и всегда)
results/tto/<run>/summary.csv      # среднее / std / SEM / CI95
results/tto/<run>/summary.json
results/tto/<run>/tto_trace.csv    # метрики на промежуточных шагах 25/50/100/200/400
results/tto/<run>/tto_curve.csv    # усреднённая кривая «шаг → Dice / фолды / время»
```

`per_case.csv` намеренно сохраняет прежнюю схему — поэтому парный Wilcoxon + BH-FDR
(`tools/analysis/compute_stats.py`) заработает по этим файлам без единой правки.

Плюс, если что-то упало — текст ошибки.

## Если не хватит памяти

Пик TTO — 3.2 GB, на 96 GB карте запас огромный. Но если вдруг:
- у координатных сетей (`inr`/`kan`/`randkan`) есть `inr_chunks` (по умолчанию 16) — можно поднять;
- не ставьте `--hd95` в том же проходе (посчитаем отдельно);
- `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`.

## Требования

- Ветка `ctcf-p3`; должны быть `utils/tto.py` и `models/KAN_IDIR/`.
- Нужен `mamba_ssm` (у вас стоит — Mamba и так обучается на этой машине).
- Профиль путей `--2` зашит в скрипт по умолчанию.
